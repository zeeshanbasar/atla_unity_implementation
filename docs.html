OTBU — Online Terrain
Barrier Update

Python/Unity simulation of the OTBU algorithm from "Robust and Near-Fuel-Optimal Landing Guidance with Online Terrain Avoidance" (Basar & Ghosh, IEEE RA-L). Implements real-time terrain barrier generation integrated with the MSS-OTALG guidance law for autonomous precision soft landing on Mars.

1. Project Overview

This repository is a Python/Unity simulation of the Online Terrain Barrier Update (OTBU) algorithm, presented in:

[OTBU] S. Z. Basar and S. Ghosh, "Robust and Near-Fuel-Optimal Landing Guidance with Online Terrain Avoidance," IEEE Robotics and Automation Letters (RA-L), 2025.

OTBU generates terrain barrier functions on the fly from a world-coordinate-registered elevation grid map (onboard downward-facing camera), and feeds them into the Multiple Sliding Surfaces – Optimal Terrain Avoidance Landing Guidance (MSS-OTALG) law from [1] to steer the spacecraft away from hazardous terrain while achieving precision soft landing.

  • MSS-OTALG [1]: base guidance law combining ZEM/ZEV optimal guidance, a terrain divert term, and a Multiple Sliding Surfaces robust correction term
  • OTBU (Algorithm 1): online update of polynomial barrier functions ρx, ρy, ρz at adaptive intervals, based on H-maxima peak detection
  • Unity terrain simulation: substitutes for an onboard depth camera; provides live elevation grid maps over TCP
  • Adaptive RK45 integration: trajectory propagation with variable step size

Terrain is validated against real Martian elevation maps from NASA's HiRISE database (DTM: "Possible Vent in Ceraunius Fossae" [15]).

3. Mathematical Foundations

All mathematics follows the paper [OTBU] and the base MSS-OTALG law from [1]. Notation is preserved from those works.

3.1 Equations of Motion

The landing site is the origin of a local East-North-Up (ENU) frame. The lander is modelled as a 3-DOF point mass (paper Eq. (1)):

\[ \dot{\mathbf{r}} = \mathbf{v}, \qquad \dot{\mathbf{v}} = \mathbf{a}_c + \mathbf{g} + \mathbf{a}_p \] \[ \dot{m} = -\frac{\|\mathbf{T}\|}{I_{sp}\,g_e}, \qquad \dot{\mathbf{a}}_c = \frac{\mathbf{a}_{cmd} - \mathbf{a}_c}{\tau} \]

where \(\mathbf{g} = [0,\,0,\,-g]^T\) (valid for low-altitude powered descent), \(\mathbf{a}_c = \mathbf{T}/m\) is the guidance acceleration command, \(\mathbf{a}_p\) is bounded perturbation acceleration, and \(\tau\) is the actuator lag time constant. A ±5% uniform noise band is applied per thrust component:

\[ T_i \sim \mathcal{U}(0.95\,T_{cmd,i},\; 1.05\,T_{cmd,i}) \]

3.2 Time-to-Go Estimation

The optimal time-to-go is computed using the method of [14]. Solving for the energy-optimal terminal time yields a quartic in \(t_{go}\):

\[ D \cdot t_{go}^4 + A \cdot t_{go}^2 + B \cdot t_{go} + C = 0 \] \[ A = -2\!\left(\|\mathbf{v}_0\|^2 + \mathbf{v}_f \cdot \mathbf{v}_0 + \|\mathbf{v}_f\|^2\right), \quad B = 12\,(\mathbf{r}_f - \mathbf{r}_0)\cdot(\mathbf{v}_0 + \mathbf{v}_f) \] \[ C = -18\|\mathbf{r}_f - \mathbf{r}_0\|^2, \quad D = \tfrac{1}{2}\|\mathbf{g}\|^2 \]

The minimum real non-negative root is \(t_{go,opt}\). As noted in the paper, this does not account for time lost to divert manoeuvres, so a 20 s buffer is added: tf = tgo_init(...) + 20.

3.3 MSS-OTALG Guidance Law

The base guidance law is MSS-OTALG [1]. The full command is (Eq. (2) of [OTBU]):

\[ \mathbf{a}_c = \frac{6}{t_{go}^2}\,\text{ZEM} - \frac{2}{t_{go}}\,\text{ZEV} + \mathbf{p}\,\frac{t_{go}^2}{12} - \boldsymbol{\Phi}\,\text{sgn}(\mathbf{s}_2) \]
\[ \text{ZEM} = \mathbf{r}_f^d - \!\left[\mathbf{r}(t) + \mathbf{v}\,t_{go} + \tfrac{1}{2}\mathbf{g}\,t_{go}^2\right], \quad \text{ZEV} = \mathbf{v}_f^d - \left[\mathbf{v}(t) + \mathbf{g}\,t_{go}\right] \]

The four terms: ZEM/ZEV provides fuel-optimal steering; \(\mathbf{p}\,t_{go}^2/12\) is the OTBU divert term; \(-\boldsymbol{\Phi}\,\text{sgn}(\mathbf{s}_2)\) is the MSS robust term.

Implementation note: The paper uses \(\text{sgn}(\mathbf{s}_2)\). This code substitutes a saturation function with boundary layer \(\varepsilon = 0.1\) to eliminate chattering: \[ \text{sat}(s_2,\varepsilon) = \begin{cases} s_2/\varepsilon & |s_2| < \varepsilon \\\\ \text{sgn}(s_2) & |s_2| \geq \varepsilon \end{cases} \]

3.4 Sliding Mode Control

Surface 1 (position error):

\[ \mathbf{s}_1 = \mathbf{r} - \mathbf{r}_f^d, \qquad \dot{\mathbf{s}}_1 = \mathbf{v} - \mathbf{v}_f^d \]

Surface 2 with virtual controller gain \(\Lambda = 2\):

\[ \mathbf{s}_2 = \dot{\mathbf{s}}_1 + \frac{\Lambda}{t_{go}}\,\mathbf{s}_1 \]

Switching gain \(\boldsymbol{\Phi}\) with \(k_1 = 0.2\), \(k_2 = 0.8\):

\[ \boldsymbol{\Phi} = k_1\,a_{p,max}\,\mathbf{I} + k_2\,\frac{|\mathbf{p}|}{12}\,t_{go}^2 \]

3.5 OTBU Algorithm — Terrain Barrier Update

Algorithm 1 (from [OTBU]). Initialised with \(\mathbf{r}_H\), \(\mathbf{r}_f^d\), \(t_{last}\), \(\delta\), \(\Delta t_1 < \Delta t_2\), \(\Theta\), \(\alpha\), \(m\).

procedure OTBU(F, r, t):
  if ‖r − rf‖ < Θ:  Δt ← Δt1  (near: faster updates)
  else:            Δt ← Δt2  (far: slower updates)

  if t − tlast ≥ Δt or t == 0:
    tlast ← t
    im    ← ExtendedMaxima(F, α)  # H-maxima transform
    s     ← GetCentroids(im)
    rpeak ← max(s)

    if rz − rHz > δ:  ρz ← rHz + δ;  b ← rH
    else:            ρz ← rpeakz + δ;  b ← rpeak

    for ν = x, y:
      if bν > rfν:
        iν ← (bν−rfν)^m / (bz−rfz)
        jν ← (bν−rfν)^m − iν·bz
        ρν ← (iν·rz + jν)^(1/m) + rfν
      elif bν < rfν:
        pν ← (rfν−bν)^m / (bz−rfz)
        qν ← (rfν−bν)^m − pν·bz
        Tν ← −rfν
        ρν ← −((pν·rz + qν)^(1/m) + Tν)
      else:
        ρν ← Inf  # canyon case
  • The vertical barrier switches from \(\mathbf{r}_H\) (orbital pre-estimate) to \(\mathbf{r}_{peak}\) (onboard camera) once below \(r_{Hz}+\delta\), minimising guidance transients
  • Lateral barriers are polynomial curves of order \(m=0.5\) passing through both \(\mathbf{b}\) and \(\mathbf{r}_f^d\)
  • Canyon case (\(b_\nu = r_{f\nu}^d\)): \(\rho_\nu = \infty\); remaining barriers still prevent a crash

3.6 Divert Repulsion Field

For each axis \(i \in \{x,y,z\}\), given signed distance \(d_i \triangleq r_i - \rho_i\) (Eq. (3) of [OTBU]):

\[ \psi_i = \frac{\ell_{2,i}}{d_i^2 + \ell_{1,i}}, \qquad \dot{p}_{r_i} = \frac{d_i\,\ell_{2,i}\,\ell_{3,i}\,e^{-\psi_i}}{(d_i^2 + \ell_{1,i})^2} \]

with \(\ell_{1,i}=1\), \(\ell_{2,i}=9500\), \(\ell_{3,i}=500\). The field vanishes both on the barrier (\(d_i \to 0\)) and far from it (\(d_i \to \infty\)), producing a smooth repulsion lobe at intermediate distances.

3.7 Composite Acceleration Command (Full)

The complete MSS-OTALG guidance command as implemented (saturation substituted for sgn):

\[ \mathbf{a}_c = \underbrace{\frac{6\,\text{ZEM}}{t_{go}^2} - \frac{2\,\text{ZEV}}{t_{go}}}_{\text{ZEM/ZEV optimal}} + \underbrace{\mathbf{p}\,\frac{t_{go}^2}{12}}_{\text{OTBU divert}} + \underbrace{-\boldsymbol{\Phi}\,\text{sat}(\mathbf{s}_2,\varepsilon)}_{\text{MSS robust}} \]

Passed through first-order actuator lag \(\tau\), then converted to thrust \(\mathbf{T} = \mathbf{a}_c \cdot m\).

4. Module Reference

main.py — Entry Point

Defines initial and terminal conditions, instantiates guidance parameters, defines the ODE right-hand side f(t, y), and drives the integrator.

SymbolValueDescription
r0[704, 145, 721] mInitial position (ENU-like frame)
v0[-55.9, 22.9, -83.8] m/sInitial velocity
m01905 kgInitial mass
rf[453, 786, 145] mTarget landing site
vf[0, 0, 0.5] m/sTerminal velocity (soft touchdown)
Tmax8 × 3100 / √3 NMaximum thrust magnitude
tau5/90 sActuator time constant
eps0.1SMC boundary layer width
Design note: The state dict is used as a closure to persist t_last and rho_last across ODE evaluations. This prevents unnecessary Unity TCP queries during RK45 sub-steps while keeping the ODE function stateless in signature.
rk45.py — Adaptive Integrator

Implements the Runge-Kutta-Fehlberg (RK45) method with adaptive step size control. Computes both 4th-order (y4) and 5th-order (y5) solutions at each step.

Local truncation error estimate and step update rule:

\[ E = \max_i |y5_i - y4_i|, \qquad h_{new} = 0.9 \cdot h \cdot \left(\frac{\text{tol}}{E}\right)^{0.2} \]

Early termination if: \(\|\mathbf{r} - \mathbf{r}_f\| < 0.1\) m, or any state element is non-finite.

tgo_min.py — Time-to-Go Estimation

Solves the quartic polynomial (§3.2) via np.roots. Returns the minimum real non-negative root as the energy-optimal \(t_{go,\min}\).

tgo_init(r0, v0, rf, vf) -> tf  # final time = t_go,opt + 20s buffer
a0.py — Initial Acceleration

Evaluates the full composite guidance law at \(t=0\) to provide a consistent initial condition for the actuator state \(\mathbf{a}_c(0)\). This avoids a step discontinuity at \(t = 0^+\).

initAcc(r0, v0, rf, vf, eps, tf, highest, t=0) -> (a0, t_last, rho_init)
barriers.py — Terrain Barrier Computation

Queries Unity terrain data via UnityTerrainController, applies h-maxima peak detection, and computes the 3D barrier surface \(\boldsymbol{\rho}(r_z)\) as described in §3.5.

barriers(r, rf, highest, rho_last, t_last, t) -> (rho, t_last)

Caching: If \(|t - t_{last}| < \Delta t\), the cached rho_last is returned without querying Unity — this is critical for performance during RK45 sub-steps.

divert.py — Repulsion Field

Computes the smooth barrier repulsion gradient \(\mathbf{p}\) from signed distance \(\mathbf{d} = \mathbf{r} - \boldsymbol{\rho}\). See §3.6 for the full derivation.

divert(r, rho) -> (d, p)
sliding.py — Sliding Surface

Computes the two SMC surfaces \(\mathbf{s}_1\) and \(\mathbf{s}_2\) (§3.4). Default decay rate \(\lambda = 2\).

sliding(r, rf, v, vf, tgo, lam=2) -> (s1, s2)
saturation.py — Boundary Layer Saturation

Vectorised saturation function (§3.4). Uses np.where for efficient elementwise evaluation over 3-vector inputs.

saturation(x, width=1.0) -> np.ndarray
show.py — Visualisation

Loads rk45_results.npz, replays the trajectory through Unity's camera controller, and plots mass and acceleration histories. Note the coordinate remap for Unity's Y-up convention:

controller.move_camera(r[i,0], r[i,2], r[i,1])
# Python y (North) → Unity Z
# Python z (Up)    → Unity Y
TerrainMap/unityTerrainController.py

TCP client for the TerrainDataExporter server, located at ./TerrainMap/unityTerrainController.py. Sends JSON-encoded commands and deserialises the returned terrain height map.

MethodDescription
send_command(command_dict)Low-level: opens socket, sends JSON, returns parsed response
move_camera(x, y, z)Move orthographic camera to world position
move_and_export(x, y, z)Atomic move + terrain export command
load_terrain_data()Read terrain_data.json from disk → {x, y, z} arrays

The terrain data schema:

{
  "x": [float, ...],    // 1D, length=resolution (world X coords)
  "y": [float, ...],    // 1D, length=resolution (world Z coords, Unity convention)
  "z": [[float, ...]]   // 2D resolution×resolution, height values
}
Note: Each call to send_command opens and closes a fresh TCP socket. This keeps the protocol simple but adds per-call overhead; connection pooling would reduce latency for high-frequency terrain queries.
TerrainMap/Assets/TerrainDataExporter.cs

Unity-side TCP server attached to a GameObject in the scene. Listens on 127.0.0.1:5555 and also provides real-time trajectory visualisation via a child LineRenderer.

TCP Commands

Trajectory Visualisation

On Start(), a child GameObject named "CameraTrajectory" is created with a LineRenderer attached. Each camera move appends the new position to trajectoryPoints and is immediately reflected in the renderer, giving a live 3D descent trace inside the Unity scene.

Inspector FieldTypeDefaultDescription
showTrajectorybooltrueToggle trajectory rendering
trajectoryColorColorRedLine colour
trajectoryWidthfloat0.5Line width in world units
trajectoryMaterialMaterialnullOverride; falls back to Sprites/Default

Thread safety: Camera movement and terrain export are deferred to Update() via flag fields — required because Unity scene objects must only be accessed from the main thread.

Race condition: Thread.Sleep(1) in move_and_export does not guarantee the camera has moved before shouldExportData = true is consumed by Update(). A ManualResetEvent handshake is required for a robust fix.

8. Running the Simulation

Step 1 — Start Unity

Open the Unity project and enter Play mode. Confirm the TCP server is running (check Console for "Server started on port 5555").

Step 2 — Run the Guidance Simulation

python main.py

This integrates the trajectory and saves results to rk45_results.npz.

Step 3 — Visualise

python show.py

Streams the trajectory back through the Unity camera and plots mass and acceleration time histories.

10. Known Limitations & Future Work

Bugs & Discrepancies with the Paper

\(T_{max}\) inconsistency (main.py): The ODE thrust cap uses T_max = 10*3100 = 31000 N (matches paper), but the \(a_{p,max}\) calculation uses 8*3100/√3 ≈ 14318 N.
Highest point hardcoded: main.py sets highest = [725, 581, 224] with a comment acknowledging it is a guess. Paper Algorithm 1 requires \(\mathbf{r}_H\) from pre-processed orbital images — should be queried via get_highest_point() at startup.
Race condition (TerrainDataExporter.cs): Thread.Sleep(1) in move_and_export does not guarantee the camera has moved before shouldExportData = true. A ManualResetEvent handshake is required.

Limitations

  • Single dominant peak assumed per terrain patch (regionProps[0] only)
  • \(\text{sgn}(\mathbf{s}_2)\) replaced with \(\text{sat}(\mathbf{s}_2, \varepsilon)\) — widens the convergence boundary layer
  • No fuel constraint enforcement in the guidance law
  • Sequential (blocking) Unity TCP calls add latency per integration step

Planned Extensions

  • Multi-peak obstacle handling
  • Successive convexification (SCvx) for fuel-optimal re-planning
  • Hardware-in-the-loop testing with IMU noise injection
  • Replace file-based terrain transfer with shared memory or UDP
